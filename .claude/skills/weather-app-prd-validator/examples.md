# Examples（输入 → 输出风格示例）

> 说明：示例用于"固定输出结构与语气"。真实校验时，请始终以用户 PRD 为准。

## 示例 1：缺少口径与验收（节选）

### 输入（PRD 片段）

```
- 需求：优化逐小时预报，让用户更直观看到降雨趋势，更准确。
- 方案：逐小时卡片增加降水概率，支持折线图。
- 验收：体验更好，信息更清晰。
```

### 输出（节选）

```markdown
## PRD 校验结论
- 就绪度：Needs work
- 阻塞项数量：P0 1 项，P1 2 项
- 总体风险：中（"更准确/更直观"缺少可测口径与验收阈值，易返工）

## 章节覆盖率（是否可评审）
| 章节 | 是否存在 | 是否可验收 | 主要缺口（一句话） |
|---|---|---|---|
| 4.1 验收标准 | 是 | 否 | 只有感受性表述，无可测阈值与用例 |
| 5.3 功能详细说明 | 是 | 否 | 缺字段口径/刷新缓存/失败兜底与交互规则 |

## 待确认信息（缺什么列什么）
- 降水概率的数据源（华风/墨迹/双源）与刷新频率
- 折线图默认展示指标（温度/降水概率/风速等）与切换交互
- 数据缺失/延迟时的兜底展示策略

## 问题清单（按 P0→P3→建议排序）
| ID | 位置/章节 | 类型 | 严重度 | 问题描述 | 风险 | 建议修正（可直接改写） |
|---|---|---|---|---|---|---|
| 1 | 4.1 验收标准 | 可测试性 | P0 | "体验更好/更清晰/更准确"不可验收 | 测试无法设计用例，开发目标不一致 | 追加可测阈值与 GWT 验收（见下方改写示例） |
| 2 | 5.3 逐小时预报 | 数据 | P1 | 未定义降水概率字段口径与单位 | 展示不一致、对比竞品口径冲突 | 写清字段（probability %）、来源、刷新与缓存策略 |
| 3 | 5.3 逐小时预报 | 交互 | P1 | 折线图的默认态/切换/无障碍未定义 | UI 反复调整 | 补充交互规则与可访问性要求（可聚焦/朗读/对比度） |

## 建议补充的 PRD 文案（可复制回 PRD）
### 1) 验收标准（示例格式）
- Given 用户进入首页且网络正常
  When 展示逐小时预报
  Then 默认展示未来 24 小时温度折线图，并在每小时节点显示降水概率（0–100%）
- Given 当前城市未来 2 小时存在降水概率 ≥ 60%
  When 用户查看逐小时预报
  Then 在时间轴上高亮对应小时段，并展示"可能降雨"提示文案（含频控：24 小时内最多提示 1 次）
- Given 数据获取失败或超时（>5s）
  When 用户查看逐小时预报
  Then 展示缓存数据 + "上次更新时间"提示，并提供重试按钮

### 2) 数据口径补充
- 数据源：华风（主）/ 墨迹（备）
- 字段：precipitation_probability（降水概率，单位 %，整数，0-100）
- 刷新频率：前台进入/下拉刷新触发，最小间隔 5 分钟
- 缓存有效期：30 分钟，过期后展示"上次更新时间"提示
```

### 输入（PRD 片段）

```
- 新增功能：开启"到点降雨提醒"，后台定时定位并推送通知提醒用户带伞。
```

### 输出（节选）

```markdown
## PRD 校验结论
- 就绪度：Not ready
- 阻塞项数量：P0 2 项，P1 1 项
- 总体风险：高（涉及后台定位与推送，合规检查表未填，上线风险高）

## 章节覆盖率（是否可评审）
| 章节 | 是否存在 | 是否可验收 | 主要缺口（一句话） |

| 6.x 隐私合规 | 否 | 否 | 后台定位+推送涉及个人信息处理，但合规表格缺失 |

| 7.x NFR | 否 | 否 | 未定义耗电/流量预算、频控策略 |

## 待确认信息（缺什么列什么）
- 后台定位的精度要求（精确/粗略）与频率
- 推送的频控策略（冷却时间、每日上限、免打扰时段）

## 问题清单（按 P0→P3→建议排序）
| ID | 位置/章节 | 类型 | 严重度 | 问题描述 | 风险 | 建议修正（可直接改写） |
|---|---|---|---|---|---|---|
| 1 | 6.x 隐私合规 | 合规 | P0 | 后台定位与推送涉及个人信息处理，但合规检查表未填/未说明采集目的与频率 | 合规/审核阻塞，上线风险 | 在 6.1/6.2 补"数据变化说明、最小化、拒绝路径、弹窗文案、频率与缓存、关闭方式" |
| 2 | 5.3 功能说明 | 流程 | P0 | 未定义触发条件、阈值、失败兜底 | 开发无法实现，测试无法验收 | 补充完整的降雨提醒规则（见模板） |
| 3 | 7.x NFR | 耗电/流量 | P1 | 未定义后台刷新预算、定位频率、失败重试与频控 | 耗电、通知打扰、投诉 | 明确后台调度策略与频控（例如仅在用户订阅且未来 2 小时降水概率≥阈值时启用） |

## 建议补充的 PRD 文案（可复制回 PRD）
### 1) 6.x 隐私合规补充
| 检查项 | 是否涉及 | 说明 |
|---|---|---|
| 定位权限 | 是 | 后台定位用于判断用户当前位置的降雨情况；精度：粗略（网络定位）；频率：每 30 分钟一次 |
| 推送权限 | 是 | 推送降雨提醒通知；用户可在设置中关闭 |
| 数据最小化 | 是 | 仅采集必要的位置信息，不上传具体经纬度，仅使用城市级别定位 |
| 用户拒绝路径 | 是 | 拒绝定位权限：降级为手动选城 + 提示；拒绝推送权限：仅展示 App 内卡片 |

### 2) 降雨提醒规则补充（5.3）
```markdown
#### 降雨提醒：需求规则
- 数据源：华风分钟级降水｜字段：未来 2 小时降水概率、预计开始时间
- 触发条件：用户已开启提醒 + 定位城市未来 2 小时降水概率 ≥ 70%
- 触发时机：后台定时检查（每 30 分钟）或前台刷新时
- 频控：同一城市 6 小时内最多 1 次；每日最多 3 次；22:00-08:00 免打扰
- 通知文案：标题"即将降雨" + 摘要"预计 {time} 开始，记得带伞"
- 点击行为：跳转至分钟级降雨详情页
- 失败兜底：定位失败 → 使用上次定位或手动选城；数据缺失 → 不触发提醒
```

## 待确认信息（缺什么列什么）

- 预警通知策略：哪些等级需要 push？频控规则？

- 预警解除后的处理策略

## 问题清单（按 P0→P3→建议排序）
| ID | 位置/章节 | 类型 | 严重度 | 问题描述 | 风险 | 建议修正（可直接改写） |
|---|---|---|---|---|---|---|
| 1 | 5.3 预警功能 | 数据 | P1 | 预警类型/等级/颜色枚举未定义 | 展示不一致，无法测试 | 引用权威枚举（国标气象预警），写明类型/等级/颜色映射 |
| 2 | 5.3 预警功能 | 流程 | P1 | 预警通知策略未定义 | 打扰用户或漏推重要预警 | 补充通知策略：哪些等级强提醒、频控、免打扰 |
| 3 | 5.3 预警功能 | 流程 | P1 | 预警解除/过期处理未定义 | 卡片残留或突然消失困惑用户 | 补充状态机：新增→生效→解除的 UI 变化 |

## 建议补充的 PRD 文案（可复制回 PRD）
### 气象预警：展示与通知规则
```markdown
#### 预警类型与等级（引用国标）
- 类型：暴雨/暴雪/寒潮/大风/高温/台风/雷电/冰雹/大雾/霾/道路结冰等
- 等级：蓝色（IV级/一般）→ 黄色（III级/较重）→ 橙色（II级/严重）→ 红色（I级/特别严重）

#### 展示规则
- 首页预警卡：展示当前城市最高等级预警，点击进入预警列表
- 预警列表：按等级降序、时间升序排列；同城多条预警全部展示
- 详情页：预警内容、发布时间、生效时间、发布单位

#### 通知策略
- 红色/橙色预警：立即推送，允许突破免打扰
- 黄色预警：推送，受免打扰限制
- 蓝色预警：仅 App 内卡片展示，不推送
- 频控：同一预警类型 6 小时内不重复推送

#### 状态机
- 新增：出现卡片 + 触发通知（按等级）
- 生效中：刷新后保持展示
- 解除/过期：卡片消失，列表移至"历史预警"（保留 24 小时）
```
```

## 输出风格要点

1. **结论先行**：先给就绪度结论，再展开细节
2. **问题分级**：严格按 P0→P1→P2→P3→建议排序
3. **可复制改写**：所有建议都应该可以直接复制到 PRD
4. **不编造**：口径/枚举/阈值不确定时，写"待确认"而非编造
5. **聚焦天气领域**：专项校验覆盖数据源、定位、预警、生活指数等天气特有维度
